# 02-03 — Pillar + sub-service dynamic route templates

## Objective

Render the 4 pillar pages and 17 sub-service pages from ONE data-driven template each,
via nested dynamic segments enumerated from the taxonomy with `dynamicParams = false`
(success criterion 2; IA-03, IA-04). Compose the 02-02 components into the D-01 (sub-
service) and D-02 (pillar) layouts, wire basic per-page metadata (D-13), and ensure every
route pre-renders on `output: export`.

This is the phase's core deliverable: after it, all 22 service routes exist and build.

## must_haves

- `app/diensten/[pillar]/page.tsx` and `app/diensten/[pillar]/[service]/page.tsx` exist,
  each with `export const dynamicParams = false;`, a correct `generateStaticParams()`,
  and an async default page that **awaits `params`** (Next 16).
- `generateStaticParams` yields exactly 4 pillar params and 17 service params, all from
  the registry (no hardcoded slug lists).
- `generateMetadata` returns `{ title: content.metaTitle, description: content.metaDescription }`
  for each route (D-13 basic only — no canonical/OG/JSON-LD).
- Pillar layout = D-02 (Hero → sub-service card grid → BrandGrid(brandsForPillar) → pillar
  FAQ → RelatedServices(other pillars) → CTABanner). Sub-service layout = D-01 (Breadcrumbs
  → ServiceHero → ServiceSteps → BrandGrid(node.brandIds, Installatie only) → ServiceFAQ(+localAngle)
  → RelatedServices(siblings) → CTABanner).
- Unknown pillar/service slug 404s (guaranteed by `dynamicParams = false`; add a `notFound()`
  guard as defense-in-depth).

taxonomy data rendered as auto-escaped React children. Open-redirect: N/A (hrefs are
registry-derived via `urlFor`). Residual risk: **none (LOW)**. Form/route hardening is Phase 5.
