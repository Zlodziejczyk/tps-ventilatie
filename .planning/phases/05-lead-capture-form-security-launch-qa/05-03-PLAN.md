<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user input → client form | Untrusted keystrokes; the client validates for UX only — the server (05-02) is authoritative |
| client form → /api/lead | Same-origin POST carrying lead PII + consent; no secret crosses here (the route holds it) |
| form copy → AVG compliance | Consent + the named-processor disclosure are the legal boundary for processing PII via GHL |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5-failsafe | DoS (usability) / Repudiation | A submit that hangs on "sending" or fails silently → a lost lead | mitigate | `submitLead` try/catch + the form maps rejection/non-OK to a visible `error` state with Bel + WhatsApp + retry (D-07); proven on preview by forcing 502/offline (05-06) |
| T-5-consent | Compliance (AVG) | Processing PII without consent / undisclosed processor | mitigate | Required consent checkbox (client) + `consent: z.literal(true)` (server, 05-02) + GoHighLevel named as verwerker in `/privacy-beleid` (LEAD-06) |
| T-5-spam | DoS / Tampering | Bot submissions | mitigate (client half) | Hidden `website` honeypot rendered here; server silently drops honeypot-filled bodies (05-02) |
| T-5-secret | Information Disclosure | Re-introducing a client-side webhook | mitigate | `lib/forms.ts` holds NO URL/secret — it only calls the same-origin route; verify-grep blocks `NEXT_PUBLIC_GHL_WEBHOOK_URL` |
| T-5-xss | Tampering | Reflected input → XSS | accept/avoid | Form values are never echoed as HTML; success/error copy is static Dutch text (keep it that way) |
</threat_model>
